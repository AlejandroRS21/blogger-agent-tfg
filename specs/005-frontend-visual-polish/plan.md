# Implementation Plan: Frontend Visual Polish & Dynamic Article Diversity

**Spec**: [spec.md](spec.md)

## 1. Technical Context

- **Frontend Framework**: Next.js 14+ (App Router).
- **Styling**: Tailwind CSS + `@tailwindcss/typography`.
- **Data Layer**: Static JSON files in `docs/` (generated by Python backend).
- **Diversity Issue**: Current generated articles follow a rigid, repetitive structure (identical headings, similar phrasing), unlike the target blogger (Javipas) who varies structure based on content needs.
- **Goal**: Implement visual polish while introducing structural diversity in generated content by refining the agents' prompt strategy.

### Unknowns & Risks
- **Agent Flexibility**: Can the current `content_generator` be easily modified to support varied structures without breaking the `html_builder`? (NEEDS CLARIFICATION)
- **Style Profiles**: Does the current `javipas_style_profile.json` contain sufficient variety in structural patterns? (NEEDS CLARIFICATION)
- **Zod Validation**: Do the current Zod schemas support varied nesting or optional sections in the HTML? (NEEDS CLARIFICATION)

## 2. Constitution Check

- **Principle I (Python Pipeline Ownership)**: Changes to article structure must be implemented in the Python agents (`aphra_blogger/agents`). (PASS)
- **Principle II (Reproducible Content Generation)**: New structural variations must be traceable to prompt templates or style profiles. (PASS)
- **Principle V (Static Delivery)**: Modified HTML structures must remain compatible with the static site generation in `docs/`. (PASS)
